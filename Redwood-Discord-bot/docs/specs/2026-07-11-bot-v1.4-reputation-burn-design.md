# Redwood Peak Bot v1.4 — Reputation + Identity Burn

**Date:** 2026-07-11
**Status:** Approved design
**Builds on:** v1.1 (identities), v1.2 (`/lookup`), the embed system.
**Scope:** Slice 3 of 5 (lookup✓ → registries✓ → **reputation+burn** → carousel → orders/ledger/budgets).

## Goal

Superiors issue commendations and write-ups that attach to a member's active cover identity; High Command can ceremonially burn a cover, which resets its standing (archived on the burned identity) and issues fresh papers while keeping the member's rank. Standing shows in `/whois` and `/lookup`.

## Data model

**New table `reputation`** — each row tied to the cover it was earned under:
- `id` uuid pk · `identity_id` uuid → identities · `discord_id` text → members · `kind` text `commendation|writeup` · `reason` text · `issued_by` text · `created_at` timestamptz default now()

Rows attach to the member's **currently-active identity** at issue time. Standing is always read for the active identity, so burning/rotating naturally resets it (old rows stay archived on the now-burned identity).

**Schema tweak:** `identities.status` check gains `burned` (now `active | retired | burned`). Run `db/schema-v1.4.sql` (drops + recreates the status check constraint, creates `reputation`, enables RLS).

## Commands

- **`/commend @member reason:<text>`** — supervisor-and-up. Files a commendation on the member's active cover. Permission is checked in-command: load the issuer's roster row and require `rankOrder(issuer.rank) ≥ rankOrder('supervisor')`. Errors if the target has no active identity.
- **`/writeup @member reason:<text>`** — supervisor-and-up. Same, `kind = 'writeup'`.
- **`/identity burn @member reason:<text>`** — **High Command only** (new subcommand in the existing `/identity` group). Marks the member's active identity `burned`, mints a fresh cover via the existing `issueIdentity`, logs a roster event. Reply is the new identity packet card titled "Cover burned — new papers filed", noting prior standing archived.

**What burn does:** resets the cover's standing (rows stay on the burned identity, not carried forward), marks the identity `burned`, issues fresh papers. **Keeps** the member's employee name and rank. Whether a stack of write-ups warrants a burn is HC's judgment — no automatic threshold.

## Display

`/whois` and `/lookup` member files gain a **Standing** field for the active cover: `✔ N commendations · ⚠ M write-ups`, plus the most recent one or two reasons. Counts are not sensitive, so they are shown to all viewers (no redaction). A burned member reads as clean because standing follows the active identity.

## Components

- `db/schema-v1.4.sql` — `reputation` table, `identities.status` `burned`, RLS.
- `src/lib/reputation.ts` — `summarizeReputation(rows): { commendations: number; writeups: number; recent: string[] }` (pure).
- `src/db/reputation.ts` — `addReputation(identityId, discordId, kind, reason, issuedBy)`, `reputationForIdentity(identityId)`, `reputationForIdentities(ids)`, `burnActiveIdentity(discordId)`.
- `src/commands/reputation.ts` — `/commend`, `/writeup` (supervisor+ check via roster rank + `rankOrder`).
- `src/commands/identity.ts` — add the `burn` subcommand (HC; the group is already `highCommandOnly: true`).
- `src/lib/lookup.ts` — `LookupData` gains `reputation: ReputationLite[]`; `buildLookupResult` member-file gains `standing: { commendations; writeups; recent }`.
- `src/db/lookup.ts` — `gatherLookup` loads reputation for the resolved identities.
- `src/lib/embeds.ts` — `whoisEmbed` gains an optional `standing` param; `lookupEmbed` member-file renders the standing line.
- `src/commands/roster.ts` — `/whois` loads standing (active identity's reputation) and passes it to `whoisEmbed`.

## Testing

- **Unit (vitest):** `summarizeReputation` (counts by kind, recent list capped); `buildLookupResult` member-file standing counts from `reputation` rows; `whoisEmbed`/`lookupEmbed` render the standing line via `.toJSON()`.
- **tsc-verified:** `db/reputation.ts`, the `gatherLookup` + `/whois` wiring, the three command handlers.
- **Live smoke test (user):** `/commend` a member → `/whois` shows it → `/writeup` → count rises → `/identity burn` → `/whois` reads clean; `/lookup` the old burned cover (HC) still shows the archived name in past identities.
