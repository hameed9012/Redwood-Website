# Redwood Peak Bot v1.6b-2 — Orders Pipeline Design

**Date:** 2026-07-12
**Status:** Draft (design)
**Builds on:** v1.6b-1 storefront (the public embed + `Place an order` button that opens a
private ticket thread) and v1.6a ledger (`ledger_entries`, `addLedgerEntry`).

## Purpose

Turn the order tickets from v1.6b-1 into a **tracked pipeline** with real state, so an
order moves through defined stages, an employee owns it, and completed work **posts its
earnings to the ledger automatically** as a white-book inflow. Until now the ticket thread
was just a conversation; this slice gives it a lifecycle and makes it pay.

## The lifecycle

```
open  →  claimed  →  fulfilled  →  done
                                     └─ ledger inflow posted (white, source='order')
        (any stage) →  cancelled     (no ledger entry)
```

- **open** — created when a customer clicks `Place an order` (v1.6b-1 already opens the
  thread; this slice also writes an `orders` row in that moment).
- **claimed** — an employee takes ownership. Records who and when.
- **fulfilled** — the work is done in ERLC; the employee sets the **amount** owed/collected.
- **done** — the order is closed out and the amount posts to the ledger as an inflow. Ledger
  write happens exactly once, on the transition into `done`.
- **cancelled** — terminal, no money moves. Available from any non-done stage.

State only ever moves forward (or to `cancelled`); it never goes backward.

## Scope

**In scope**
- `orders` table (schema-v1.6b.sql).
- Writing an `open` order row when the storefront order thread is created (hook into the
  existing v1.6b-1 `handleOrderButton`).
- In-thread action buttons on the order ticket: **Claim**, **Fulfilled**, **Done**, **Cancel**
  — each advancing state and re-rendering the order status card.
- `/order` slash command (in-thread): `claim`, `fulfill amount:<>`, `done`, `cancel`,
  `status` — a keyboard-driven mirror of the buttons for reliability.
- On `done`: `addLedgerEntry(amount, 'inflow', 'white', 'Order #<n> — <summary>', 'order', actorId)`.
- Order status card embed reflecting current stage, claimer, amount, customer.
- Pure state-machine module (unit-tested) governing legal transitions.

**Out of scope**
- Editing an order's line items / itemized pricing (single total amount only for now).
- Website surfacing of orders (a later reporting slice; the ledger already carries the money).
- Refunds / partial payments / outflows tied to orders.
- Assigning orders to someone other than the claimer.

## Architecture

### `src/lib/orders.ts` — pure, unit-tested
- `OrderStatus = 'open' | 'claimed' | 'fulfilled' | 'done' | 'cancelled'`.
- `Order` interface: `{ id, seq, customerId, threadId, status, claimedBy, amount, summary, createdAt, ... }`.
- `TRANSITIONS: Record<OrderStatus, OrderStatus[]>` — the legal-move table encoding the
  lifecycle above.
- `canTransition(from, to): boolean` and `assertTransition(from, to): { ok } | { ok:false, error }`.
- `requiresAmount(to): boolean` — `fulfilled`/`done` need a positive amount set.
- `orderButtons(status)` → the ActionRow of buttons valid for the current stage (e.g. no
  `Claim` once claimed; only `Done`/`Cancel` after fulfilled).
- `orderCardEmbed(order)` → tone-colored status card (info while active, success on done,
  denied on cancelled).
- id constants: button custom ids carry the order id, e.g. `rw_order_claim:<orderId>`.

### `src/db/orders.ts` — I/O, tsc-verified + live smoke test
- `createOrder(customerId, threadId, summary)` → inserts `open`, returns `Order` (with a
  human sequence number `seq` from a table sequence).
- `getOrder(orderId)`, `getOrderByThread(threadId)`.
- `updateOrderStatus(orderId, to, patch)` — persists the new status plus fields
  (`claimed_by`, `amount`, timestamps).

### `src/interactions/orders.ts` — I/O
- `handleOrderAction(interaction, cfg)` — parses `rw_order_<action>:<orderId>` button ids,
  loads the order, validates the transition via `assertTransition`, applies it, and on `done`
  calls `addLedgerEntry`. Re-renders the card. Ephemeral errors on illegal moves ("This order
  is already claimed", etc.). `fulfill` opens a small amount modal (`rw_order_amount:<orderId>`)
  since a button can't collect a number.
- Registered in the v1.6b-1 `routeButton` / `routeModal` dispatch (prefix `rw_order_`).

### `src/commands/order.ts` — `/order` subcommands
- Thin wrappers over the same db + state-machine functions, resolving the order from the
  current thread (`getOrderByThread(interaction.channelId)`). HC not required — any active
  roster member can claim/fulfill/done; `cancel` restricted to the claimer or High Command.

### Hook into v1.6b-1
- `handleOrderButton` (storefront) gains one line after creating the thread:
  `await createOrder(interaction.user.id, thread.id, summary)`, then posts the order card +
  action buttons into the thread.

## Data model — schema-v1.6b.sql

```sql
create sequence if not exists public.order_seq;

create table if not exists public.orders (
  id          uuid primary key default gen_random_uuid(),
  seq         bigint not null default nextval('public.order_seq'),
  customer_id text not null,
  thread_id   text not null,
  status      text not null default 'open'
              check (status in ('open','claimed','fulfilled','done','cancelled')),
  claimed_by  text,
  amount      integer check (amount is null or amount > 0),
  summary     text not null default '',
  created_at  timestamptz not null default now(),
  claimed_at  timestamptz,
  done_at     timestamptz
);
alter table public.orders enable row level security;  -- service key only, like the rest
```

No change to `ledger_entries`; orders post into it via the existing `source='order'` tag,
so `/ledger summary` and reconciliation already understand them.

## Error handling

- **Illegal transition** (e.g. `Done` before `fulfilled`, or acting on a `cancelled` order) →
  ephemeral in-voice rejection; nothing persisted, no ledger write.
- **`Done` without an amount** → blocked; prompt to run `fulfill` (set the amount) first.
- **Double-`done` / double-post** → the ledger write is gated on the `fulfilled → done`
  transition succeeding; once `done`, `Done` is not offered and `assertTransition` rejects it,
  so earnings can never post twice.
- **Ledger write fails on done** → the status change is rolled back to `fulfilled` (or not
  committed) so a `done` order always has its money on the books; ephemeral error to the actor.
- **Action on a thread with no order row** → ephemeral "no order is attached to this thread."

## Testing

**Unit (vitest)** — `src/lib/orders.ts`:
- `canTransition`: every legal edge true; a spread of illegal edges false (backward moves,
  anything out of `done`/`cancelled`, `open→done`).
- `requiresAmount('fulfilled')` / `('done')` true; others false.
- `orderButtons(status)`: correct button set per stage (no Claim when claimed; Done+Cancel
  after fulfilled; none when done/cancelled).
- `orderCardEmbed`: tone/color per status; shows amount + claimer when present.

**tsc** — full bot typecheck for db/interaction/command modules + the v1.6b-1 hook.

**Live smoke test (user)** — post storefront; click `Place an order`; in the thread: `Claim`
→ `Fulfilled` (set amount) → `Done`; confirm the order card updates each step and
`/ledger summary` shows the amount as a white inflow tagged `order`; separately test `Cancel`
posts nothing.

## Deployment / config note

- Run `schema-v1.6b.sql` in Supabase (creates `order_seq` + `orders`).
- No new env vars. Same thread permissions from v1.6b-1 (Create Private Threads / Manage
  Threads) cover this slice.
- Ships after v1.6b-1, which owns the button/modal routing this slice extends.
