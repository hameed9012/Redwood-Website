# Redwood Peak Bot v1.1 — Identity Packets + Shift System

**Date:** 2026-07-11
**Status:** Approved design
**Builds on:** v1 (roster/ranks/divisions/positions, security suite, persona, `/say`, server logger)

## Goal

Add two paired features to the bot, plus the three website portal pages that read what they write:

1. **Identity Packets** — `/identity create` becomes the single onboarding ritual (replacing `/hire`): the runner supplies the member's real **roleplay name** (from their application); the bot generates only a disposable **legal cover identity**, onboards the member (rank role + nickname = the RP name), and files the packet. Re-running rotates the cover while keeping the RP/employee name.
2. **Shift System** — the "ankle monitor": `/shift start/log/party/end/report/status` records on-duty periods, structured incidents (people/plates/badges that will feed `/lookup` later), a movement account, and optional witness-dossier reports.
3. **Website** — build the **Personnel**, **Operations Log**, and **Witness Dossiers** portal pages, reading bot-written data server-side.

## Architecture

- **Stack:** TypeScript + discord.js v14, shared Supabase (service-role, server-side only), self-hosted on Ubuntu via pm2, Node 20 (CommonJS build — see v1).
- **Interaction/logic split:** pure logic in unit-tested modules (vitest); thin Discord/DB adapters verified by `tsc` and live smoke test. A Discord gateway cannot run in CI.
- **Website reads (Approach A):** the bot owns all writes with the service key. The three portal pages are Next.js **server components** calling server-only loaders (`lib/portal/*.ts`) that read Supabase with the service key and return only **site-safe** columns. Page-level access reuses the existing client-side `RequireTier` tier gating.
- **Security invariant:** sensitive cover data (SSN, legal name, DOB, ID number) is **never** sent to any browser — it lives only in Discord, viewable by High Command via `/identity view`. The service key is server-only in both the bot and the website; it is never imported into a client component.

## Data model (Supabase)

Builds on the existing `members` table (holds stable `employee_name`, rank, divisions, positions, status). Five new tables. RLS **on**, no anon policies — only the bot's service key writes; the website reads server-side.

### `identities` — disposable cover packets (one member → many over time; exactly one `active`)
| Column | Notes |
|---|---|
| `id` | uuid pk |
| `discord_id` | → members |
| `legal_name` | full civilian name; rotates |
| `dob` | date; ages ~21–55 |
| `ssn` | `XXX-XX-XXXX`, valid-looking |
| `id_number` | driver's-license style (letter + digits) |
| `blood_type` | e.g. `O+` |
| `next_of_kin` | name + relation |
| `issued_at` | timestamptz |
| `status` | `active` \| `retired` |
| `retired_at` | timestamptz, null while active |

First `/identity create` takes the member's real roleplay name (a required `name` option, from their application) and writes it to `members.employee_name` (stable) — the bot does NOT invent it. Rotation retires the current identities row and inserts a fresh `active` one; `employee_name` untouched.

## Feature 1 — Identity Packets

All under the `/identity` command group, **High Command only**.

- **`/identity create @user name:<rp-name> rank:<rank>`** — onboarding ritual, replaces `/hire`. On a member with no identity: take the supplied RP name as `employee_name`, generate the legal cover, set nickname to the RP name, assign the rank role via the existing `applyRosterChange` path, insert the `active` identities row, redraw the roster. Errors if the member already has an identity (directs to `/identity rotate`).
- **`/identity rotate @user`** — re-roll for an existing member: retire the current cover, issue a fresh legal name + numbers, keep employee name / nickname / rank / standing.
- **`/identity view @user`** — render the member's current packet as a company-voice formatted document (paperwork styling). High-Command-only; cover details are sensitive in-fiction.

### Generation — `src/lib/identityGen.ts` (pure, unit-tested)

All randomized for fictional characters; never mapped to real people (real *format*, random *values*).

- **employee_name** — NOT generated; supplied manually from the member's application (their real RP name), stable, becomes nickname
- **legal_name** — a *different* full civilian name; rotates
- **dob** — random, ages ~21–55
- **ssn** — `XXX-XX-XXXX`, valid-looking ranges
- **id_number** — driver's-license style
- **blood_type**, **next_of_kin** (name + relation)

`/hire` is removed from the command set. Other roster commands (`/promote`, `/demote`, `/setrank`, `/division`, `/position`, `/dismiss`, `/whois`) unchanged.

## Feature 2 — Shift System

All under the `/shift` command group. **Self-serve** for any active member on the roster (not High-Command-gated). Structured data is captured via **slash-command options**, not modals (modals cap at 5 inputs and can't repeat).

- **`/shift start`** — open a shift tied to the caller's currently-active identity. Errors if they already have an open shift, or aren't on the roster.
- **`/shift log summary:<> location:<> [name] [cover_name] [plate] [badge] [role]`** — record an incident during the shift. If party fields are filled, the primary party is captured inline. Repeatable.
- **`/shift party name:<> [cover_name] [plate] [badge] [unit] [role]`** — add another party to the caller's most recent incident this shift.
- **`/shift end movements:<text>`** — close-out: store the movement account, mark the shift `closed`, reply with a company-voice summary (duration, incident count). *"Shift closed. Your movements have been recorded."*
- **`/shift report subject:<> body:<>`** — optional witness-dossier write-up tied to the shift; publishes to Witness Dossiers.
- **`/shift status`** — show the caller's open shift and what's been logged, or that they're off duty.

Pure logic (shift-duration math, "has an open shift?", "most recent incident for this shift", party validation) lives in unit-tested modules; Discord/DB adapters are thin.

**Deliberate cut:** no enforced shift nagging, no random check-ins. It is a records tool, not a timer.

## Feature 3 — Website portal pages

Each is a Next.js server component calling a server-only loader in `lib/portal/*.ts` (service key, server-side). Page access reuses client-side `RequireTier`. Cosmetic `<Redacted>` bars remain for atmosphere. No cover secrets are ever sent to the browser.

1. **Personnel** (`/portal/personnel`, employee tier) — live org chart: roster grouped by rank; each person's employee name, rank, divisions, positions. Web mirror of the bot roster. No cover data.
2. **Operations Log** (`/portal/operations-log`, employee tier) — closed shifts as a timeline: date, who, duration, movement account, and each incident (summary, location, plates/names as a manifest).
3. **Witness Dossiers** (`/portal/witness-dossiers`, high-command tier) — persons of ongoing interest: `reports` plus aggregated `incident_parties` (outside civilians, officers by badge, plates) with the incidents each appears in. A web preview of the future `/lookup`.

**New website env var:** `SUPABASE_SERVICE_KEY` in Vercel (server-only; never imported client-side). Added to deploy notes.

## Testing

- **Unit (vitest):** `identityGen` (name/SSN/DL/DOB format + shape), shift-duration math, open-shift detection, most-recent-incident lookup, party validation, portal loader data-shaping (rows → view model).
- **tsc-verified:** `/identity` and `/shift` command handlers, DB repositories, server-side portal loaders.
- **Live smoke test:** run the commands on the server; confirm the three pages render.

## Out of scope for v1.1 (later slices)

- `/lookup` query command itself (we store the structured party data now; querying comes later)
- Full identity **burn** mechanics (standing/commendation loss); `/identity rotate` covers new-papers
- Standalone serial / vehicle **registries** (plates are logged in incidents; registry commands later)
- Orders / ledger / budgets
- The media-carousel control command (queued separately)
