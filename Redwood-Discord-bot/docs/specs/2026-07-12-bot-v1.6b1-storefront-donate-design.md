# Redwood Peak Bot v1.6b-1 — Storefront + Donations Design

**Date:** 2026-07-12
**Status:** Approved (design)
**Builds on:** v1.6a ledger (`ledger_entries`, `addLedgerEntry`, `computeBalances`, `formatMoney`, `source` tagging)

## Purpose

Give Redwood Peak a public face for taking work and receiving money:

1. A High-Command command posts a **public storefront embed** to a channel — who we
   are, that orders are open — with two buttons: **Place an order** and **Donate**.
2. **Place an order** opens a **private ticket thread** and pings High Command. The
   order is taken by hand in-thread for now. (Tracked order state — claim/fulfill/
   done → auto-post earnings — is a separate next slice, **v1.6b-2**.)
3. **Donate** opens a **modal** asking for a sum and whether to include a name or stay
   anonymous. On submit the amount is recorded to the ledger (white-book inflow,
   `source='donation'`) and logged to a private donations channel. Donations are
   **private by default**; a public acknowledgement fires **only** for amounts at or
   above the public threshold ($50,000).

This slice writes only to the existing ledger — no new economy tables. Its main new
infrastructure is **button and modal interaction routing**, which the bot does not yet
have.

## Scope

**In scope**
- `/storefront post channel:<#channel>` (HC only) — posts the public embed with buttons.
- `rw_order` button → creates a private thread off the storefront channel, adds the
  clicker, pings High Command, posts an intake prompt. Ephemeral confirmation to the clicker.
- `rw_donate` button → shows the donation modal.
- `rw_donate_modal` submit → validate amount → ledger white inflow (`source='donation'`)
  → private donations-log post → ephemeral thank-you → public acknowledgement iff amount ≥ threshold.
- Button/modal routing in the interaction handler.
- New optional config `channelDonations` (`CHANNEL_DONATIONS`). If unset, the private
  donations-log post is skipped (everything else still works).

**Out of scope (v1.6b-2 and later)**
- Tracked order lifecycle (claim / fulfill / mark done), order DB table, earnings auto-posting.
- Any website surfacing of orders or donations.
- Editing/re-posting an existing storefront embed (just post a fresh one).

## Architecture

### Interaction routing (the new part)

Today `Events.InteractionCreate` early-returns unless `interaction.isChatInputCommand()`.
It also **defers every command** (`deferReply` ephemeral) before dispatch. Buttons and
modals need different handling and must be routed *before* the command path:

- **Buttons** are matched by a `customId` prefix. `rw_donate` must call
  `interaction.showModal(...)` as its **first** response — it therefore must **not** be
  deferred. `rw_order` does slower work (thread creation), so it defers ephemerally first,
  then uses `editReply`.
- **Modal submits** (`rw_donate_modal`) defer ephemerally, then `editReply`.

Routing shape added to the handler, ordered so each interaction type is handled once:

```ts
if (interaction.isButton()) return routeButton(interaction, cfg);
if (interaction.isModalSubmit()) return routeModal(interaction, cfg);
if (!interaction.isChatInputCommand() || !interaction.guild) return;
// ...existing command path unchanged...
```

`routeButton` / `routeModal` live in `src/interactions/storefront.ts` and own their own
defer/try-catch (mirroring the command path's 50013-aware error handling), so `index.ts`
stays a thin dispatcher.

### Components

**`src/lib/storefront.ts`** — pure, unit-tested. No discord.js clients, only builders/parsers.
- `STOREFRONT` id constants: `ORDER_BUTTON = 'rw_order'`, `DONATE_BUTTON = 'rw_donate'`,
  `DONATE_MODAL = 'rw_donate_modal'`, field ids `DONATE_AMOUNT = 'amount'`,
  `DONATE_NAME = 'name'`.
- `PUBLIC_DONATION_THRESHOLD = 50000`.
- `storefrontEmbed()` → tone `info` embed (who we are + orders open).
- `storefrontButtons()` → `ActionRowBuilder<ButtonBuilder>` with the two buttons.
- `donateModal()` → `ModalBuilder` with a required amount field and an optional name field
  (blank/empty ⇒ anonymous).
- `parseDonation(amountStr: string)` → `{ ok: true; amount: number } | { ok: false; error: string }`.
  Strips `$`, commas, and spaces; accepts whole positive dollars only; rejects zero,
  negatives, decimals, and non-numeric input.
- `donationLogLine(amount, name)` and `donationPublicLine(amount, name)` → strings for the
  private log and the public shout-out (name or "an anonymous benefactor").

**`src/interactions/storefront.ts`** — I/O, tsc-verified + live smoke-tested.
- `handleOrderButton(interaction, cfg)` — defer ephemeral; create a private thread
  (`ChannelType.PrivateThread`) on the storefront channel named `order-<username>`; add the
  clicker; post an intake message pinging `cfg.roleForRank['high-command']`; `editReply` an
  ephemeral "your ticket is open" pointer.
- `handleDonateButton(interaction)` — `interaction.showModal(donateModal())` (no defer).
- `handleDonateModal(interaction, cfg)` — defer ephemeral; read + `parseDonation`; on error
  `editReply` a deadpan rejection; on success `addLedgerEntry(amount, 'inflow', 'white',
  reason, 'donation', interaction.user.id)`; if `cfg.channelDonations`, post
  `donationLogLine` there; if `amount >= PUBLIC_DONATION_THRESHOLD`, post
  `donationPublicLine` in the storefront channel; `editReply` ephemeral thanks.
- `routeButton` / `routeModal` — prefix dispatch + shared error handling.

**`src/commands/storefront.ts`** — `/storefront post` (HC only), `channel` option
(`addChannelOption`). Sends `{ embeds: [storefrontEmbed()], components: [storefrontButtons()] }`
to the target channel, then `editReply` a confirmation. Registered in `src/commands/index.ts`.

**`src/lib/config.ts`** — add optional `channelDonations?: string` from `CHANNEL_DONATIONS`
(same pattern as `channelLog`; not added to `REQUIRED`).

### Data flow — donation

```
click Donate → modal (amount, optional name)
  → submit → parseDonation
      ok?  no → ephemeral rejection, stop
           yes ↓
      addLedgerEntry(amount,'inflow','white','Donation …','donation',userId)
      channelDonations set? → post private log line
      amount ≥ 50,000? → post public acknowledgement in storefront channel
      → ephemeral thank-you to donor
```

The reason string carries provenance, e.g. `Donation from <name>` or `Donation (anonymous)`,
so `/ledger summary` stays legible.

### Data flow — order

```
click Place an order → defer ephemeral
  → create private thread on storefront channel (order-<username>)
  → add clicker; post intake prompt pinging High Command
  → ephemeral pointer to the thread
```

## Error handling

- **Bad donation amount** → ephemeral deadpan rejection; nothing written. No public/log post.
- **Ledger write fails** → error bubbles to the route's try-catch → ephemeral "it has been
  noted"; no log/public post (they run only after a successful write).
- **`CHANNEL_DONATIONS` unset** → skip the private log; donation still recorded; donor still
  thanked. (Intentional graceful degrade.)
- **Thread creation fails** (missing Create Private Threads / Manage Threads, or channel is
  itself a thread) → 50013-aware ephemeral message via the shared handler.
- **Threshold public post fails** (deleted channel/perms) → caught and swallowed so the
  donor still gets thanked and the ledger entry stands.

## Testing

**Unit (vitest)** — `src/lib/storefront.ts`:
- `parseDonation`: `"50000"`, `"$50,000"`, `" 1000 "` → ok; `"0"`, `"-5"`, `"1.5"`, `"abc"`,
  `""` → error.
- Builders `.toJSON()`: storefront embed has expected title/tone color; button row has two
  buttons with ids `rw_order` / `rw_donate`; modal has amount (required) + name (optional)
  with correct custom ids; `donateModal().data.custom_id === 'rw_donate_modal'`.
- `donationLogLine` / `donationPublicLine`: named vs anonymous wording; `formatMoney` applied.
- `PUBLIC_DONATION_THRESHOLD === 50000`.

**tsc** — full bot typecheck for the I/O modules and the new routing in `index.ts`.

**Live smoke test (user, on server)** — deploy note: bot role needs **Create Private
Threads** + **Manage Threads** (+ Send Messages in the storefront channel). Then:
`/storefront post`; click Donate with a small sum (private only, logged if configured);
click Donate with ≥ $50,000 (public shout-out appears); click Place an order (thread opens,
HC pinged); `/ledger summary` shows both donations as white inflow.

## Deployment / config note

Add to the server `.env`:
```
CHANNEL_DONATIONS=<private donations-log channel id>   # optional
```
No schema changes — reuses `ledger_entries` from v1.6a.
