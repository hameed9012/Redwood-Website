# Redwood Peak Bot v1.2 — `/lookup` (the police database)

**Date:** 2026-07-11
**Status:** Approved design
**Builds on:** v1.1 (shifts + incident_parties), the embed system.
**Scope:** First of a five-slice backlog (lookup → registries → reputation+burn → carousel → orders/ledger/budgets). This spec covers **only `/lookup`**.

## Goal

A single `/lookup query:<text>` that searches everything the org has logged and returns a consolidated file — an internal member's file or an outsider's dossier — clearance-gated so non-High-Command members get the operational surface but cannot unmask a cover identity.

## Architecture

- **Read-only.** No schema changes; `/lookup` queries the existing `members`, `identities`, `incidents`, `incident_parties` tables.
- **Pure brain, thin I/O** (the pattern used throughout): `src/lib/lookup.ts` classifies the query and assembles the result view-model with redaction applied; `src/db/lookup.ts` runs the searches; `src/commands/lookup.ts` orchestrates; `src/lib/embeds.ts` renders. Pure logic is vitest-unit-tested; the Discord/Supabase I/O is tsc-verified and smoke-tested live.
- Reply is **ephemeral**, available to **all active members**.

## Behavior

### Query classification — `classifyQuery(raw): Query`
- A Discord mention (`<@123>` / `<@!123>`) or a bare 17–19-digit snowflake → `{ kind: 'member', discordId }`.
- Anything else → `{ kind: 'text', text }` (trimmed).

### Search
For a `text` query, search (case-insensitive):
- `members.employee_name` — partial contains
- `identities.legal_name` — partial contains (any status, active or retired)
- `incident_parties.name` and `incident_parties.cover_name` — partial contains
- `incident_parties.plate` and `incident_parties.badge` — exact (case-insensitive)

For a `member` query, load that member directly.

### Subjects and results — `buildLookupResult(query, data, viewerIsHC): LookupResult`

`LookupResult` is one of:

- **`member-file`** — the query resolved to one internal member (via @mention, employee name, or a cover legal name belonging to one of their identities). Contents:
  - Always: employee name, rank, standing (active/dismissed).
  - Incidents: incidents they logged while on shift, plus incidents where their cover name or a plate on their file appears.
  - HC-only: the current cover packet (legal name, DOB, SSN, ID number, blood type, next of kin), the list of past/retired identities, and the fact that a looked-up cover maps to this employee.
  - Non-HC: cover details render redacted (`▓▓▓▓`); if the subject was reached by looking up a **cover name**, the employee is NOT revealed — the result is an `outsider-dossier`-style "cover on file · ▓▓▓▓" rather than a member file (non-HC cannot cross the cover→employee bridge).

- **`outsider-dossier`** — the query matched parties in incidents but no internal member (a civilian, an officer badge, a plate). Contents: the identifier, inferred role (civilian/officer/witness/other from the party rows), every incident they appear in (summary, location, date, and the employee who logged it), and other identifiers seen in those same incidents (co-occurring plates/badges/names).

- **`disambiguation`** — more than one distinct subject matched. A short list (up to 8) of `label + kind` for the user to re-query more specifically. No guessing.

- **`not-found`** — nothing matched.

### Redaction rule (the clearance line)
`viewerIsHC` is computed in the command from the caller's roles (the high-command role id from config, via the existing `hasHighCommandRole` check).
- **High Command:** full detail — cover↔employee link, full packet, past/burned identities.
- **Everyone else:** operational surface only. Cover packet fields redacted; a cover-name lookup never resolves to the employee behind it.

## Components

- `src/lib/lookup.ts` — `classifyQuery`, `buildLookupResult`, and the `Query` / `LookupResult` types. Pure.
- `src/db/lookup.ts` — `searchMembersByName`, `searchIdentitiesByLegalName`, `searchParties` (name/cover/plate/badge), `incidentsForMember`, `incidentsForParties`, `identitiesForMember`. Raw-row returns.
- `src/commands/lookup.ts` — the `/lookup` command; `highCommandOnly: false`; computes `viewerIsHC`; gathers data; builds result; replies with `lookupEmbed`.
- `src/lib/embeds.ts` — `lookupEmbed(result: LookupResult): EmbedBuilder`, switching on `result.kind` (member file / outsider dossier / disambiguation / not found). Info tone; redacted values shown as `▓▓▓▓`.

## Testing

- **Unit (vitest):**
  - `classifyQuery` — mention, raw snowflake, and free text.
  - `buildLookupResult` — with synthetic rows and `viewerIsHC` both ways: member file assembled correctly; HC sees legal name/SSN and the cover→employee link; non-HC gets `▓▓▓▓` and a cover-name lookup does NOT reveal the employee; outsider dossier aggregates incidents + co-occurring identifiers; multiple matches → disambiguation; nothing → not-found.
  - `lookupEmbed` via `.toJSON()` — correct title/fields per kind, redaction markers present for non-HC.
- **tsc-verified:** `src/db/lookup.ts`, `src/commands/lookup.ts`, the embed wiring.
- **Live smoke test (user):** run `/lookup` on a plate, a badge, a cover name, and an @member as both an HC and a non-HC account; confirm the redaction holds.

## Command inventory after v1.2

Added: `/lookup`. Everything else unchanged.

## Out of scope (later slices)

Serial/vehicle registries (v1.3) will add `serial`/`plate` registry rows that `/lookup` can later search — but this slice only reads what the shift system already logs. Reputation/burn, carousel, orders/ledger/budgets untouched. No new tables.
